# 迁移策略

## 总览

当前项目**没有正式迁移系统**。数据库结构由 `dsl.app` 启动时调用 `create_tables(Base)` 初始化。

这意味着 Koda 目前的数据库策略更接近“开发期自动建表”，而不是“生产期可回滚迁移”。

## 当前现状

### 已有能力

- 新数据库首次启动时会自动创建缺失表
- 不需要手工执行建表脚本
- 对本地快速迭代很方便

### 明确限制

- `create_all` 不会自动修改已有表结构
- 新增列、改列名、改类型、删列都不会被自动处理
- 没有迁移版本表
- 没有升级与回滚历史

## 当前开发期做法

如果你只是在本地开发，数据库结构发生变化时，建议按影响等级处理。

### 只新增表

通常只需要：

1. 更新 ORM 模型
2. 启动应用
3. 让 `create_tables(Base)` 自动补齐新表

### 修改已有表结构

例如新增列、改类型、改约束，这时需要人工处理。

常见做法：

1. 备份 `data/dsl.db`
2. 手动执行 SQLite 迁移
3. 或在确认数据可丢弃时，重建开发数据库

!!! warning "重要"
    不能把 `create_tables(Base)` 误认为“自动迁移器”。它只能创建缺失表，不能安全升级已有表。

## 推荐变更清单

每次改动 `dsl/models/` 时，至少回答下面几个问题：

1. 这次是新增表，还是修改已有表
2. 现有 `data/dsl.db` 是否还能兼容
3. 前端类型和 API 响应是否需要同步更新
4. 文档中的数据模型页是否需要更新

## 未来建议

如果项目继续沿着需求卡片平台演进，建议尽快引入正式迁移工具，例如 Alembic。

### 未来迁移体系应该覆盖

- 版本化升级脚本
- 回滚路径
- SQLite 与未来数据库方言的一致策略
- 结构变更与数据修复分离

### 建议的落地顺序

1. 引入 Alembic
2. 固化基线版本
3. 把后续所有 `dsl/models/` 变更纳入迁移脚本
4. 在 CI 或发布流程中加入迁移检查

!!! note "待补充"
    当前仓库还没有 `alembic/` 目录或迁移命令。本页记录的是现状和建议，便于后续补齐工程化能力。
